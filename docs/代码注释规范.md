# Toonflow 代码注释规范

本文档是 `AGENTS.md` 中“注释规范”的详细版，用于沉淀更完整的实践约束。`AGENTS.md` 只保留短规则，避免主规则文件过长、分散模型注意力。

## 1. 总原则

- 代码优先自解释，注释只补充代码本身不容易表达的信息。
- 注释的主要职责是解释：
  - 为什么必须这样写
  - 这段逻辑依赖什么约束
  - 这段逻辑在防什么边界情况或历史问题
- 注释不是命名不清、结构混乱的替代品。

## 2. 什么情况下应该写注释

- 业务约束不直观，单看函数名和变量名仍无法理解原因。
- 需要兼容历史数据、脏数据、旧状态或旧文件格式。
- 涉及异步状态流转、文件落盘时序、轮询时序、重试时序。
- 需要兼容 Electron、VM、Node、浏览器等不同运行环境。
- 对接第三方模型、第三方 API 或供应商时，存在特殊限制或怪异行为。
- 代码看起来像“可以删掉”，但实际承担了防回归、防误删、防重复执行等作用。

## 3. 什么情况下不应该写注释

- 代码本身已经足够清晰，读者无需额外解释即可理解。
- 注释只是逐字翻译代码动作。
- 注释重复了类型、函数名、变量名已经明确表达的信息。
- 注释内容空泛，不能帮助未来维护者做判断。

不推荐的例子：

```sh
# 设置 ROOT_DIR
ROOT_DIR=...
```

```ts
// 遍历列表
for (const item of items) {
```

```py
# 打印结果
print(result)
```

## 4. 推荐的注释风格

- 简短、具体、可验证。
- 优先说明原因和约束，再说明动作。
- 用语要能帮助维护者判断“能不能删、能不能改、为什么不能直接简化”。

推荐的例子：

```ts
// 这里不能直接信任 o_image.state，因为历史任务可能已标记完成，但底层文件仍然损坏。
```

```ts
// vendor 代码运行在 VM 中，不能假设存在浏览器全局对象。
```

```sh
# GUI 运行时实际读的是 Electron 用户目录，因此这里需要先同步仓库内置资源。
```

## 5. Shell 脚本注释规范

- 适用于 `scripts/` 下的脚本。
- 默认应具备以下三类注释：
  - 文件级注释：说明脚本职责、适用场景、是否会影响进程、数据库或运行时目录。
  - 函数级注释：说明函数存在原因、关键副作用、关键边界条件。
  - 关键步骤注释：说明参数解析、环境切换、资源同步、数据库处理、进程清理等阶段性动作。
- 以下 shell 逻辑应优先补注释：
  - 切换 Node / nvm 环境
  - 同步 Electron 运行时资源
  - 清理进程、删除文件、改动数据库
  - 区分当前仓库与其他本地副本

## 6. 通用代码注释规范

- TypeScript / JavaScript / Python / Shell 均遵循同一原则：只为不直观的逻辑写注释。
- 注释如果超过几行，先判断是否应拆函数、重命名、重组结构。
- 对新增逻辑，如果其存在明显副作用或隐藏约束，应同步补充注释，而不是只修改代码。

## 7. 注释与重构的关系

- 如果一段逻辑必须依赖很长的段落注释才能理解，优先考虑：
  - 重命名变量 / 函数
  - 拆出更小的函数
  - 调整代码顺序
  - 消除重复分支
- 只有在这些手段仍不足以表达清楚时，再使用注释解释。

## 8. Toonflow 项目中的高优先级注释场景

- 图片 / 视频生成链路中的状态流转。
- 历史坏图、坏视频、脏数据库记录的兼容处理。
- GUI 与 Electron 用户目录的同步逻辑。
- vendor 在 VM 中运行时的环境限制。
- 会影响用户运行时数据库、成功资产或历史记录的清理逻辑。

## 9. 一句话标准

好的注释应该回答：

“为什么这段代码不能被一个不了解上下文的人轻易删掉、改掉或简化掉？”

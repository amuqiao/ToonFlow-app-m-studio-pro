# AI短剧平台中的两种核心架构：资产先行 vs 分镜先行

> **文档职责**  
> 归纳 AI 短剧 / 漫剧创作平台中两种常见的流程架构，即“资产先行”与“分镜先行”，并分析它们在数据模型、工作流、系统治理和产品体验上的差异。  
>
> **适用场景**  
> 适用于产品架构设计、工作流设计、Agent 体系设计、平台对标分析，以及在评估 Toonflow、LibTV 等创作平台时建立统一的分析框架。  
>
> **阅读目标**  
> 帮助读者明确：不同平台虽然都在做 AI 短剧创作，但其底层组织方式可能完全不同；并理解这种差异会如何影响一致性、复用性、生产效率和产品形态。  
>
> **目标读者**  
> 适用于产品经理、架构设计者、后端开发、前端开发、Agent 工作流设计者，以及需要研究 AI 内容生产平台核心结构的协作者。  
>
> **维护规范**  
> 本文档优先沉淀稳定的方法论，不绑定单一产品版本。新增内容时，应围绕“核心对象、流程顺序、系统影响、适用场景”展开，不写成零散功能罗列；如后续平台形态出现新的混合架构，应以“新增架构类型”方式扩展，而不是打碎现有两类主结构。

## 结论

AI 短剧平台常见的核心架构可以抽象成两类：

- **资产先行**
- **分镜先行**

两者都能完成“从剧本到视频”的创作链路，但系统真正围绕的中心对象不同：

- **资产先行**：先把角色、场景、道具等视觉基础建好，再让分镜和视频去调用这些资产
- **分镜先行**：先快速拆镜头并形成分镜图，再在分镜和视频过程中逐步补足参考与细节

这不是交互层的小差异，而是会影响：

- 数据模型设计
- 工作流顺序
- Agent 分层方式
- 复用能力
- 一致性控制
- 工业化生产能力

## 总体对比图


## 一、什么是资产先行架构

### 1. 定义

资产先行架构的核心思想是：

**在进入分镜和视频前，先把角色、场景、道具等视觉基础对象独立出来，并将其作为后续创作的统一参考源。**

也就是说，系统默认认为：

- 分镜不是孤立镜头
- 分镜是对既有资产的调用、组合和编排

### 2. 典型流程

资产先行通常长这样：

```text
原著 / 剧本
  ↓
资产提取
  ↓
角色 / 场景 / 道具生成
  ↓
导演规划
  ↓
分镜生成
  ↓
视频生成
  ↓
导出
```

### 3. 系统的核心对象

在这种架构下，平台通常把以下对象建成一等公民：

- 项目
- 剧本
- 资产
- 分镜
- 视频轨

其中，资产层是中轴，不是附属能力。

### 4. 优势

- 更容易保证角色一致性
- 场景和道具可以跨镜头复用
- 更适合长链路、多集、批量生产
- 更容易做流程校验，例如“该分镜缺少角色资产”
- 更适合 Agent 分层协作

### 5. 代价

- 前置步骤更多
- 首次创作启动更重
- 对新手来说更像“先搭生产系统，再开始创作”

## 二、什么是分镜先行架构

### 1. 定义

分镜先行架构的核心思想是：

**先快速把故事拆成镜头，再围绕这些镜头去生成分镜图、视频片段，并在过程中逐步补足参考和细节。**

也就是说，系统默认认为：

- 创作的中心是镜头
- 参考素材服务于镜头，而不是镜头服务于资产系统

### 2. 典型流程

分镜先行通常长这样：

```text
剧情描述 / 剧本
  ↓
分镜脚本
  ↓
分镜图
  ↓
视频片段
  ↓
局部回修 / 补参考
  ↓
导出
```

### 3. 系统的核心对象

在这种架构下，平台通常更围绕这些对象组织：

- 项目
- 画布节点
- 分镜单元
- 视频片段

角色图、参考图、素材图可能存在，但不一定构成独立、强建模的资产层。

### 4. 优势

- 上手更快
- 更适合探索式创作
- 更容易快速出第一版结果
- 更符合导演直接做镜头的心智模型

### 5. 代价

- 长项目中一致性更难治理
- 资产复用能力通常较弱
- 很多问题会拖到分镜或视频阶段才暴露
- 后期更依赖人工回修而不是系统级约束

## 三、两类架构的关键差异

### 1. 中心对象不同

- **资产先行**：核心是“资产”
- **分镜先行**：核心是“镜头 / 分镜”

### 2. 系统约束位置不同

- **资产先行**：约束前置，先把世界建好
- **分镜先行**：约束后置，先开始做镜头，发现问题再修

### 3. 复用机制不同

- **资产先行**：复用角色、场景、道具
- **分镜先行**：复用镜头模板、提示词、局部参考

### 4. 工业化程度不同

- **资产先行** 更像生产系统
- **分镜先行** 更像创作工作台

## 四、阶段输入 / 输出对比

| 维度 | 资产先行 | 分镜先行 |
| --- | --- | --- |
| 起点输入 | 剧本、原著、项目设定 | 剧情描述、剧本、参考图、参考视频 |
| 第一关键中间层 | 资产层 | 分镜层 |
| 中间产物 | 角色、场景、道具、衍生资产 | 分镜脚本、分镜图、镜头单元 |
| 分镜的角色 | 调用资产并组织镜头 | 直接承载创作主流程 |
| 视频的输入 | 分镜 + 资产 | 分镜 + 参考 |
| 一致性控制 | 更依赖系统资产约束 | 更依赖创作者手工回修 |
| 适合的项目类型 | 长流程、批量、多集、工业化 | 快速试作、探索式创作、镜头优先 |

## 五、为什么这是重要的架构设计问题

这类差异不是“按钮顺序不同”，而是会直接决定平台后续怎么长。

### 1. 决定数据库和业务模型

如果你选择资产先行，系统就要把：

- 角色
- 场景
- 道具
- 资产与分镜的关联

都建成稳定对象。

如果你选择分镜先行，系统更可能先强化：

- 分镜脚本
- 分镜图
- 视频片段
- 节点之间的画布关系

### 2. 决定 Agent 的拆法

- **资产先行** 更适合拆成：
  - 剧本 Agent
  - 资产 Agent
  - 导演规划 Agent
  - 分镜 Agent
  - 视频 Agent
- **分镜先行** 更适合拆成：
  - 创作 Agent
  - 分镜 Agent
  - 图像 Agent
  - 视频 Agent
  - 画布编排 Agent

### 3. 决定用户体验

- **资产先行**：更稳，但更重
- **分镜先行**：更快，但更松

### 4. 决定后续产品边界

平台未来是往“工业化生产系统”走，还是往“创作者画布平台”走，很大程度上从这里就已经决定了。

## 六、用 Toonflow 和 LibTV 来理解这两类架构

### 1. Toonflow 更接近资产先行

Toonflow 的公开链路和代码结构都显示：

- 先处理原著和事件
- 再生成故事骨架、改编策略、剧本
- 再提取资产、生成资产
- 再做导演规划、分镜、视频

它显式建模了资产层，并在后续通过资产与分镜关联来驱动生产。

### 2. LibTV 更接近分镜先行

从公开资料看，LibTV 更强调：

- 剧本 / 剧情输入
- 分镜脚本
- 分镜图
- 视频片段
- 批量生产与单条回修

它当然也支持参考图、角色三视图、参考视频等能力，但公开表达重点并不在“先建立资产系统”，而在“先把镜头生产起来”。

### 3. 最直观的对照

可以用一句话概括：

- **Toonflow：先把世界建好，再拍**
- **LibTV：先开始拍，再逐步补世界**

## 七、并不存在绝对优劣

这两类架构并没有绝对谁更先进，只是适合的问题不同。

### 资产先行更适合

- 角色一致性要求高
- 项目周期长
- 多集短剧 / 漫剧
- 团队协作
- 需要复用资产和沉淀生产资料

### 分镜先行更适合

- 快速试验故事
- 快速验证镜头风格
- 创作者个人工作流
- 先求出结果，再逐步打磨

## 八、对 AI 短剧平台设计的启示

如果要设计一套成熟的 AI 短剧平台，真正要决定的不是“先做哪个页面”，而是：

**平台到底把什么当成创作的中枢。**

如果中枢是资产，系统会走向生产化。  
如果中枢是分镜，系统会走向创作化。

很多平台最终会走向混合形态，但即使是混合形态，也通常会有一个主导方向。

因此，在讨论产品路线时，最应该先回答的问题不是：

- 要不要做角色库
- 要不要做分镜表
- 要不要做画布

而是：

**这套系统到底是“资产驱动分镜”，还是“分镜驱动参考”。**

## 九、可用于后续分析的判断问题

以后评估任何一款 AI 短剧平台时，可以先问这 6 个问题：

1. 角色 / 场景 / 道具是否是独立业务对象？
2. 分镜是否必须显式引用这些对象？
3. 平台是先让用户建资产，还是先让用户拆镜头？
4. 一致性问题是前置解决，还是后置回修？
5. 中间层最重的是资产层，还是分镜层？
6. 这套系统更像工业化生产线，还是创作工作台？

这 6 个问题基本可以快速判断它更接近哪一种架构。

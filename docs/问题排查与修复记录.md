> **文档职责**  
> 记录 Toonflow 项目中排查成本高、容易复发、跨多层链路的问题，并沉淀对应的根因判断、修复方式与复现/回归要点。  
>
> **适用场景**  
> 适用于 GUI 开发、Electron 运行时、后端路由、Agent 工作流、媒体生成链路、vendor 适配、历史脏数据处理等问题的排查与回顾。  
>
> **阅读目标**  
> 帮助维护者在遇到相似现象时，快速判断问题归属、缩小排查范围、复用已有修复方案，并明确是否需要重启 GUI、重试任务或清理历史数据。  
>
> **目标读者**

>
> **维护规范**  
> 只记录“值得复用”的问题，不记录一次性、低价值、纯操作失误的问题。每条记录至少应包含：现象、根因、影响范围、修复方案、是否需要重启/重试、相关文件。若问题仅与历史脏数据有关，应明确标注“不是代码缺陷”；若问题修复后需要用户执行额外动作，也应明确写出。

## 1. 故事骨架提示“已保存”，但工作台与复审读取为空

### 现象

- 聊天区提示“故事骨架已保存，请在右侧工作台查看。”
- 右侧工作台读取不到 `storySkeleton`
- 复审结果提示骨架内容缺失

### 根因

- 脚本 Agent 的旧执行链路只把产出写进 memory，没有真正写入工作区数据库。
- 技能文档要求模型输出“已保存”的确认文案，但旧逻辑缺少真正的落库动作，因此聊天区提示与工作台数据不是同一条链路。

### 影响范围

- 故事骨架阶段
- 改编策略阶段
- 剧本阶段的工作区读取
- 任何依赖 `planData` 的复审/展示逻辑

### 修复方案

- 为脚本 Agent 补充真实写入工具。
- 在 Agent 结束后解析 `<storySkeleton>`、`<adaptationStrategy>`、脚本条目并落库。
- 修正 `setPlanData` 路由的数据结构错误。

### 是否需要重启 / 重试

- 需要重启 GUI，使新的 Agent / route 代码生效。
- 修复前已经“生成成功”的骨架通常需要重新生成一次，因为旧结果大概率并未真正写库。

### 相关文件

- [tools.ts](/Users/admin/Downloads/Code/Toonflow-app-master/src/agents/scriptAgent/tools.ts)
- [index.ts](/Users/admin/Downloads/Code/Toonflow-app-master/src/agents/scriptAgent/index.ts)
- [setPlanData.ts](/Users/admin/Downloads/Code/Toonflow-app-master/src/routes/scriptAgent/setPlanData.ts)
- [getPlanData.ts](/Users/admin/Downloads/Code/Toonflow-app-master/src/routes/scriptAgent/getPlanData.ts)

### 备注

- 这类问题的判断关键是区分“模型说已保存”和“数据库真的写入成功”。

## 2. OpenAI 图片生成后 `sharp` 报 `Input file contains unsupported image format`

### 现象

- 批量生成图片后，资产页或造景页出现大量缩略图报错。
- 日志中出现：

```txt
[OSS] 生成缩略图失败: Error: Input file contains unsupported image format
```

### 根因

- 旧的 OpenAI vendor 图片能力没有真正实现，可能返回空结果或无效内容，导致保存到磁盘的并不是合法图片。
- 部分生成路由调用 `aiImage.save(...)` 时没有 `await`，缩略图流程可能在文件尚未完整落盘时就开始读取。
- 页面在扫描历史图时会把旧的坏图片记录一起读出来，放大了报错量。

### 影响范围

- 资产生成
- 塑角造景
- 历史图片列表
- 缩略图生成

### 修复方案

- 为 OpenAI vendor 补充实际图片生成逻辑。
- 在图片生成路由中改为 `await aiImage.save(...)`，确保文件先完整落盘再进入后续流程。

### 是否需要重启 / 重试

- 需要重启 GUI，使更新后的 `data/vendor` 生效。
- 新任务需要重新生成。
- 历史坏图不会自动修复，必要时需要单独清理失败记录或坏文件。

### 相关文件

- [openai.ts](/Users/admin/Downloads/Code/Toonflow-app-master/data/vendor/openai.ts)
- [generateAssets.ts](/Users/admin/Downloads/Code/Toonflow-app-master/src/routes/assetsGenerate/generateAssets.ts)
- [batchGenerateImageAssets.ts](/Users/admin/Downloads/Code/Toonflow-app-master/src/routes/assetsGenerate/batchGenerateImageAssets.ts)
- [getAllAssets.ts](/Users/admin/Downloads/Code/Toonflow-app-master/src/routes/cornerScape/getAllAssets.ts)
- [oss.ts](/Users/admin/Downloads/Code/Toonflow-app-master/src/utils/oss.ts)

### 备注

- 如果“当前新图生成成功”但页面仍刷大量同类错误，优先怀疑是历史坏图被扫描，不要误判为当前新任务失败。

## 3. OpenAI 视频生成报错：`Blob is not defined`

### 现象

- 视频制作工作台生成视频时失败。
- 失败原因显示 `Blob is not defined`。

### 根因

- OpenAI 视频上传逻辑使用了浏览器环境下的 `Blob`。
- 但 `data/vendor/*.ts` 运行在 VM 沙盒中，不应假设存在浏览器全局对象。
- 当前 VM 注入了 `fetch`、`FormData` 等能力，但没有注入 `Blob`。

### 影响范围

- OpenAI 视频模型
- 参考图上传
- 视频任务创建阶段

### 修复方案

- 将视频上传实现改为 Node 兼容写法，使用 `form-data + Buffer`，不再依赖浏览器 `Blob`。
- 保持修复范围局限于 OpenAI vendor，不扩大到全局 VM 环境。

### 是否需要重启 / 重试

- 需要重启 GUI，使新的 vendor 代码同步到 Electron 运行时目录。
- 失败的视频任务需要重新发起。

### 相关文件

- [openai.ts](/Users/admin/Downloads/Code/Toonflow-app-master/data/vendor/openai.ts)
- [vm.ts](/Users/admin/Downloads/Code/Toonflow-app-master/src/utils/vm.ts)

### 备注

- 这类问题本质是“运行环境假设错误”，不是内容问题，也不是模型参数问题。

## 4. OpenAI 视频生成报错：`Inpaint image must match the requested width and height`

### 现象

- 分镜图已经成功生成，但视频生成失败。
- OpenAI 返回：

```txt
Inpaint image must match the requested width and height
```

### 根因

- 视频请求已经根据工作台选择的分辨率映射为固定尺寸，例如 `1280x720` 或 `1792x1024`。
- 但作为 `input_reference` 上传的参考图仍是原始尺寸，未先缩放到目标视频尺寸。
- OpenAI 要求参考图尺寸与请求尺寸完全一致，因此直接拒绝。

### 影响范围

- OpenAI 视频模型
- 参考图驱动的视频生成
- 视频制作工作台

### 修复方案

- 在 OpenAI vendor 中，上传 `input_reference` 之前先按目标视频尺寸执行缩放，再上传缩放后的参考图。

### 是否需要重启 / 重试

- 需要重启 GUI，使新的 vendor 代码生效。
- 失败的视频任务需要重新生成。

### 相关文件

- [openai.ts](/Users/admin/Downloads/Code/Toonflow-app-master/data/vendor/openai.ts)
- [generateVideo.ts](/Users/admin/Downloads/Code/Toonflow-app-master/src/routes/production/workbench/generateVideo.ts)

### 备注

- 这里的目标尺寸来自视频工作台选择的分辨率，不是随意的默认值。

## 5. `dev-gui-manager.sh restart` 偶发失败，原因是 Node 回落到 25.x

### 现象

- 用户曾执行过 `nvm use 24`，但后续再次 `restart` 时仍报 `engines` 不兼容。
- 日志提示当前 Node 为 `25.x`，而项目要求 `<25`。

### 根因

- `nvm use 24` 只对当前 shell 会话生效。
- 旧的 GUI 启动脚本没有在自身内部显式读取 `.nvmrc` 并切换到项目要求版本，因此新终端或新上下文中会回落到系统默认 Node。

### 影响范围

- `start`
- `restart`
- `electron-rebuild`
- `yarn dev:gui`

### 修复方案

- 在 GUI 启动脚本中显式读取 `.nvmrc`，加载 `~/.nvm/nvm.sh` 并执行 `nvm use`。
- 保持改动范围在启动脚本内，不修改业务代码。

### 是否需要重启 / 重试

- 修复脚本后，后续直接使用原有 `start/restart` 命令即可。
- 不再要求用户每次手工执行 `nvm use 24`。

### 相关文件

- [start-dev-gui.sh](/Users/admin/Downloads/Code/Toonflow-app-master/scripts/start-dev-gui.sh)
- [dev-gui-manager.sh](/Users/admin/Downloads/Code/Toonflow-app-master/scripts/dev-gui-manager.sh)
- [package.json](/Users/admin/Downloads/Code/Toonflow-app-master/package.json)
- [.nvmrc](/Users/admin/Downloads/Code/Toonflow-app-master/.nvmrc)

### 备注

- 这类问题属于环境稳定性问题，不是 Electron 本身故障。

## 6. `checkVideoStateList` 连续刷日志

### 现象

- 视频生成过程中，日志里频繁出现：

```txt
POST /api/production/workbench/checkVideoStateList 200 ... - 41
```

### 根因

- 这是前端对视频异步任务状态的正常轮询。
- 接口会周期性查询目标 `videoIds` 中是否有记录已进入 `生成成功` 或 `生成失败`。

### 影响范围

- 视频制作工作台
- 视频状态更新
- 前端任务卡片刷新

### 修复方案

- 当前不属于必须修复的问题。
- 若后续需要优化，可从前端轮询退避、只在存在“生成中”任务时轮询、页面不可见时降频等方向着手。

### 是否需要重启 / 重试

- 不需要。

### 相关文件

- [responseFormat.ts](/Users/admin/Downloads/Code/Toonflow-app-master/src/lib/responseFormat.ts)

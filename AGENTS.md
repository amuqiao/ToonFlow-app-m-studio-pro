# Toonflow 仓库协作规范

本文件用于约束 Codex 及其他代码代理在本仓库中的行为。

## 1. 作用范围

- 本文件位于项目根目录，作用于整个仓库。
- 如果更深层目录存在新的 `AGENTS.md`，则更深层规则优先。
- 用户在当前对话中直接提出的要求，优先级高于本文件。

## 2. 仓库定位

- 这是 Toonflow 的桌面端 / 后端仓库，不是前端源码仓库。
- 本仓库主要包含：
  - `src/`：Express、Agent、数据库、生成链路、路由
  - `scripts/`：Electron 启动、开发辅助、构建脚本
  - `data/web`：内置前端构建产物
  - `data/vendor`：模型供应商运行时代码
- 前端源码位于单独仓库 `Toonflow-web`。
- 如果需求涉及真正的前端源码改动，先判断是否应该在 `Toonflow-web` 中处理，不要在本仓库臆造前端源码结构。

## 3. 沟通方式

- 默认使用中文回复用户。
- 解释问题时优先明确问题归属：
  - 内容/工作流问题
  - 后端逻辑问题
  - vendor 适配问题
  - Electron 运行时问题
  - 用户数据 / 历史数据问题
- 不要泛泛而谈，要明确指出是哪一层出了问题。
- 需要重启服务、重新同步资源、重新进入某个工作流步骤时，要明确告诉用户。

## 4. 个人协作偏好

- 优先最小改动，不做无关重构。
- 优先修根因，不靠“吞异常”“关日志”“绕过去”掩盖问题。
- 在没有明确要求时：
  - 不主动新增脚本
  - 不主动新增文件
  - 不主动引入新抽象层
- 默认禁止新增文件。
- 只有在现有文件无法合理承载修改，且新增文件确实能显著降低复杂度或风险时，才允许新增文件。
- 如果新增文件是必要的，先确认是否能通过修改现有文件解决。
- 对高风险修改，先分析，再动手。

## 5. 修改前的分析要求

- 在开始改代码前，先判断问题属于哪一类：
  - 代码缺陷
  - 运行环境问题
  - 数据污染 / 历史脏数据问题
  - 模型配置或 vendor 接入问题
  - 前端展示问题
- 如果问题主要来自历史数据或运行时数据，不要一上来改代码。
- 如果问题来自异步生成链路，先理清：
  - 数据库状态如何流转
  - 文件何时落盘
  - 前端何时轮询
  - 当前报错是新任务还是历史任务触发的

## 6. 环境约定

- 使用 Node `24.x`
- 使用 Yarn `1.x`
- 需要时先加载 shell 环境：
  - `source ~/.zshrc`

## 7. 常用命令

- 类型校验：
  - `yarn lint`
- 后端开发：
  - `yarn dev`
- GUI 开发：
  - `yarn dev:gui`
- GUI + Vite 前端联调：
  - `yarn dev:gui-vite`
- 推荐的 GUI 生命周期命令：
  - `bash scripts/dev-gui-manager.sh start`
  - `bash scripts/dev-gui-manager.sh stop`
  - `bash scripts/dev-gui-manager.sh restart`
  - `bash scripts/dev-gui-manager.sh status`

## 8. 校验要求

- 只要改了 TypeScript / 后端代码，至少运行：
  - `yarn lint`
- 只要改了 shell 脚本，还要运行：
  - `bash -n <script>`
- 改了以下内容后，通常需要提醒用户重启 GUI 才会生效：
  - `data/vendor`
  - `data/web`
  - `data/skills`
  - Electron 启动逻辑
  - 运行时资源同步逻辑
- 如果无法执行校验，要明确说明原因。

## 9. Electron 运行时与仓库文件的关系

- Electron GUI 运行时数据不直接读仓库，而是读 Electron 用户目录。
- macOS 默认目录：
  - `~/Library/Application Support/Electron/data`
- GUI 启动时会把仓库中的以下目录同步到运行时目录：
  - `assets`
  - `models`
  - `skills`
  - `vendor`
  - `web`
- 因此修改仓库文件后，如果是运行时资源，通常需要重启 GUI。

## 10. Vendor 代码规则

- `data/vendor/*.ts` 运行在 VM 沙盒中，不要默认它是浏览器环境。
- 改 vendor 代码时：
  - 优先使用 Node 兼容写法
  - 不要默认存在浏览器全局对象，如 `window`、`document`、`Blob`
  - 改动前检查 `src/utils/vm.ts` 里实际注入了什么能力
- vendor 改动应尽量局部化，不要影响其他供应商。

## 11. 数据与清理规则

- 用户数据、运行时数据库、成功素材都属于高风险对象。
- 涉及以下动作时，默认必须先征求用户确认，再执行：
  - 修改运行时数据库
  - 清理历史记录
  - 删除图片 / 视频 / 分镜 / 资产文件
  - 批量删除或批量重置
- 未经用户明确要求，不要：
  - 删除成功资产
  - 重置数据库
  - 替换用户运行时数据
  - 批量清理历史记录
- 涉及清理时优先：
  - 小范围
  - 可预览
  - 可回退
  - 明确作用目标

## 12. 媒体生成链路规则

- 处理图片 / 视频生成问题时，重点检查：
  - `o_image`
  - `o_video`
  - `o_assets`
  - `o_storyboard`
- 不要默认 `state = 已完成` 就代表底层文件一定有效。
- 如果页面在扫描历史图、历史视频、历史分镜，要先确认：
  - 这是当前新任务导致的报错
  - 还是历史坏文件导致的报错
- 避免把“当前生成成功”和“历史数据报错”混为一谈。

## 13. 不要做的事

- 不要使用破坏性 git 命令，例如：
  - `git reset --hard`
  - `git checkout -- <file>`
- 不要删除与当前任务无关的文件。
- 不要在没有明确需要时增加新的脚本或工具文件。
- 不要因为一个局部问题引入大范围架构改造。

## 14. 注释规范

- 代码优先自解释，注释只补充原因、约束和边界。
- 只有非直观逻辑才需要注释，不要求新增代码全部带注释。
- 涉及历史兼容、脏数据、异步时序、环境差异、第三方特殊行为时，应优先补注释。
- shell 脚本默认要有文件级注释和关键步骤注释。
- 禁止低信息量注释，例如“设置变量”“循环参数”“打印结果”。
- 如果需要很长注释才能看懂，先优化命名、拆函数、调结构。
- 详细版见 `docs/代码注释规范.md`。

## 15. 期望的工作方式

- 先分析，再修改。
- 先定位层级，再给方案。
- 优先最小修复。
- 明确说明是否需要：
  - 重启服务
  - 重新打开 GUI
  - 重新生成图片 / 视频
  - 回到上游流程重做
